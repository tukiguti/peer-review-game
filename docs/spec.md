# 査読ゲーム 実装仕様

この仕様は zikken リポジトリの `playground/peer-review-game-plan/` が正本。ここはその写し。


---

# 01. コンセプトとルール

## コアループ

```
┌─ 発表者の手番 ────────────────────────────────┐
│ 1. 抽選   スロット風に3リールが回り、カードが3枚決まる      │
│ 2. 準備   30秒（スキップ可）                          │
│ 3. 発表   60秒。3枚に沿った研究をそれらしく語る            │
│ 4. 査読   発表者以外が端末を回して秘密投票 Accept/Reject   │
│ 5. 判定   過半数 Accept → 採択                       │
└──────────────────────────────────────────┘
        ↓ 次の発表者へ（全員が1回発表 = 1ラウンド）
```

## プレイ人数・時間

- **3〜8人**（過半数判定が成立する最小人数が3人）
- 既定ラウンド数: 全員1巡 × 2周（設定で1〜3周）
- 1手番およそ2〜3分。5人×2周で25分前後。

## 1手番の詳細

### 1. 抽選（スロット風演出）
- 3リールが回転し、左→中→右の順に停止して以下が確定する。
  - **①分野カード**（Field）: 何を研究するか
  - **②手法カード**（Method）: どうやって調べるか
  - **③制約カード**（Constraint）: 研究に課される無茶ぶり
- 演出の詳細は [03_slot-production.md](#)。
- **引き直し（リロール）**: 各プレイヤーは1ゲーム中に **2回** まで、任意の1リールだけを回し直せる。使い切ったら以降は不可。手札の運ゲー感を薄め、「この制約なら語れる」という選択の余地を残すため。

### 2. 準備（30秒・スキップ可）
- 発表者が構想を練る時間。「準備完了」でスキップできる。
- 画面には3枚のカードと、語るべき観点のヒントを出す。
  - タイトル / 新規性（なぜ今までなかったのか）/ 手法（どう検証するか）/ 想定結果 / 限界

### 3. 発表（既定60秒）
- タイマーが走る。設定で 30 / 60 / 90 / 120秒。
- 残り10秒で警告色＋カウントダウン音。
- 早く終わったら「発表終了」で投票に進める。

### 4. 査読（パスプレイ秘密投票）
- 発表者を除く全員が、1人ずつ端末を受け取って投票する。
  - 画面: 「**田中さんの番です**」→ 端末を渡す →「準備OK」→ `Accept` / `Reject` の2択 → 「次の人へ渡してください」
  - 直前の人の投票は次の人に見えないこと（画面に残さない）が絶対条件。
- 任意で**査読コメント**を一言だけ添えられる（"Reviewer #2 の一言"）。結果画面でまとめて公開され、笑いどころになる。

### 5. 判定と得点

| 対象 | 条件 | 得点 |
|------|------|------|
| 発表者 | 過半数が Accept（**採択**） | **+2** |
| 発表者 | 全員が Accept（**満場一致採択**） | **+3**（+2の上位、加算ではない） |
| 発表者 | 過半数に届かず（**不採択**） | 0 |
| 査読者 | **多数派側**に投票していた | **+1** |

- 「過半数」= 投票者数の半分より多い。同数（例: 4人中2:2）は **不採択**。査読は疑わしきは落とす。
- 査読者の加点があるため、全員が惰性で Accept すると全員が+1を取り合って差がつかず、逆に空気を読まずに Reject すると失点する。**発表の出来を素直に評価するのが最適解**になるよう設計している。

### 結果画面
- 採択/不採択のスタンプ演出（`ACCEPTED` / `REJECTED`）
- 誰が Accept / Reject したかを公開（秘密なのは投票中だけ）
- 査読コメント一覧
- 現在の得点表

## ゲーム終了
- 全ラウンド終了後、最終得点のランキングを表示。
- 最多得点者が **最優秀論文賞**。最も多く Reject を出した人に **Reviewer #2 賞**、満場一致採択を取った人に **話術賞** 等のネタ称号を出す。

## 設定項目（セットアップ画面）

- プレイヤー名の登録（3〜8人）
- 周回数（1〜3）
- 発表時間（30/60/90/120秒）
- 準備時間あり/なし
- 引き直し回数（0〜3、既定2）
- 使用するデッキ（真面目寄せ / ネタ寄せ / 全部）※ [02_cards.md](#) 参照
- 設定は localStorage に保存し、次回起動時に復元する。

## 設計上の判断メモ

- **AI査読者は入れない**: 元の発想どおり人間だけで完結させる。APIキー管理もサーバも不要になり、その場でURLを開けば遊べる。
- **オンライン対戦にしない**: 対面で「語る」ことが体験の中心なので、1画面を回す方式が合っている。秘密投票のパスプレイだけ丁寧に作れば成立する。
- **カードは3枚固定**: 4枚以上にすると語る難度が跳ね上がり、テンポが落ちる。

# 02. カードデッキ設計

3種類のデッキから1枚ずつ引く。カードは JSON（`src/data/cards.json`）で持ち、実装から文言を分離する。

## データ構造

```jsonc
{
  "field":      [{ "id": "f001", "text": "大規模言語モデル", "tone": "serious" }],
  "method":     [{ "id": "m001", "text": "ランダム化比較試験(RCT)", "tone": "serious" }],
  "constraint": [{ "id": "c001", "text": "被験者は3人まで",       "tone": "serious" }]
}
```

- `tone`: `serious`（真面目寄せ）/ `neta`（ネタ寄せ）。セットアップの「使用デッキ」設定でフィルタする。
- 文言は短く（全角14文字以内が目安）。リールに収まり、一目で読めることを優先する。

## ③制約カード（Constraint）— 課される無茶ぶり

**この山が笑いの核。** 発表者はこの制約を無視できず、必ず言い訳込みで語る必要がある。

### serious
被験者は3人まで / 予算は1万円 / 実験期間は一晩 / 先行研究が1本も存在しない / データは全て手作業で収集 / 再現実験ができない / 統計的有意差が出なかった / 倫理審査に落ちた前提で語れ / 使える機材は自分のノートPCのみ / 被験者は全員が身内 / 締切は明日 / 実験装置が途中で壊れた / 効果量が極めて小さい / 対照群が用意できない / 実験は全て遠隔で行われた

### neta
査読者の1人は自分の指導教員 / 研究費は自腹 / 共著者が全員音信不通 / 実験データを1つ消してしまった / タイトルに「深層」を必ず入れろ / 図表は1枚も作れない / この研究の社会的意義を3回言え / 中間発表で既に一度落ちている / ライバル研究室が3日前に同じ論文を出した / 学部1年生でも分かるように話せ / 「新規性はどこ？」に必ず答えろ / なぜか予算だけ1億円ある / 実は再現性が無いと薄々気づいている / 被験者は全員が猫 / 発表中に「限界」と3回言え

## 枚数の目安

| デッキ | serious | neta | 計 |
|--------|---------|------|-----|
| 分野 | 20 | 20 | 40 |
| 手法 | 20 | 12 | 32 |
| 制約 | 15 | 15 | 30 |

同一手番で同じカードが重複しないよう、各デッキから独立に1枚ずつ引く（デッキ間の重複は起こり得ない）。
同じゲーム中に同じカードが再登場するのは許容する（山札を使い切る設計にはしない）が、**直近3手番で出たカードは抽選から除外**してマンネリを避ける。

# 03. 抽選のスロット風演出

カードの抽選は「3枚めくる」ではなく **スロット** として見せる。ここが手番の掴みになる。

## 画面構成

```
┌──────────────────────────────────────────┐
│              🎰 テーマ抽選                  │
│                                          │
│   ┌────────┐ ┌────────┐ ┌────────┐        │
│   │ 分野    │ │ 手法    │ │ 制約    │        │  ← 各リールの上に見出し
│   ├────────┤ ├────────┤ ├────────┤        │
│   │ ～～～  │ │ ～～～  │ │ ～～～  │        │  ← 上に流れる（ブラー）
│   │ 味噌の  │ │ 目視で  │ │ 予算は  │        │  ← 中央=当選ライン
│   │ 熟成    │ │ カウント│ │ 1万円   │        │
│   │ ～～～  │ │ ～～～  │ │ ～～～  │        │
│   └────────┘ └────────┘ └────────┘        │
│      止まった    止まった    回転中          │
│                                          │
│        [  LEVER  ]   引き直し: 残り2       │
└──────────────────────────────────────────┘
```

## 演出仕様

| 項目 | 内容 |
|------|------|
| 回転方向 | 縦（上→下にカードが流れる）。3リールが同時に回り出す |
| 回転速度 | 約 20 カード/秒。速すぎると文字が読めず「回っている」感が出ないため、あえて文字が識別できる限界の速度にする |
| モーションブラー | 回転中は `filter: blur(1.5px)` 程度＋上下グラデーションマスクでフェードさせる |
| 停止順 | **左 → 中 → 右** の順に、0.4秒間隔で自動停止 |
| 停止演出 | 各リールが止まる瞬間に 1〜2コマ分の跳ね返り（overshoot → 戻る、120ms のイージング） |
| 停止音 | 「カコン」1音。3リール目だけ音を変えて確定感を出す |
| 制約リール（3枚目） | **最後に止まる**＝一番おいしいカードを最後に見せる。停止前に一瞬だけ減速する「引っかかり」を入れる |
| 確定後 | 3枚が中央に集まってカード表示に切り替わり、`ズームイン + 軽い発光`。ここで初めて全文が読める |
| レバー | 画面下の `LEVER` ボタン。押下時に短い振動（`navigator.vibrate(30)`、対応端末のみ）と「ジャキン」音 |

## 引き直し（リロール）

- 確定後、各カードの下に `🔄 引き直し` ボタンが出る（残り回数がある場合のみ）。
- 押すと **そのリールだけ** が再回転して停止する。他の2枚は動かさない。
- 残り回数を画面に常時表示。0になったらボタンを非表示にする。

## サウンド

- 効果音は Web Audio API で生成するか、短い mp3 を数点だけ同梱する（外部CDN不可・self-contained）。
- 全体ミュートのトグルを常時右上に置く。飲み会でうるさい場合に即切れること。

## 実装上の注意

- 回転は `requestAnimationFrame` で位置を進める。CSS animation で長時間回すと停止位置の制御が難しい。
- リールの中身は「実カード配列をシャッフルしたもの」をループさせ、**停止させたいカードが中央に来る位置で減速を開始**する。乱数で当選カードを先に決めてから、そこに向けて止める（後付けで結果を決めない）。
- 低スペック端末でカクつく場合に備え、設定に「演出オフ（即決定）」を用意する。

# 04. 技術構成

## スタック

| 項目 | 選定 | 理由 |
|------|------|------|
| ビルド | Vite | 起動が速く、GitHub Pages への静的出力が容易 |
| 言語 | TypeScript (strict) | カード・投票・得点の状態を型で守る |
| UI | React 19 | 画面がフェーズ遷移の塊なので、状態駆動が素直に書ける |
| 状態管理 | `useReducer` 1本 | 外部ライブラリ不要。ゲーム状態は単一の store で足りる |
| スタイル | CSS Modules | 依存を増やさない |
| 永続化 | localStorage | 設定のみ。ゲーム進行の途中保存はしない |
| サーバ | **なし** | 1画面パスプレイなので完全に静的 |
| デプロイ | GitHub Pages (Actions) | push で自動デプロイ |

## ゲーム状態（型の骨格）

```ts
type Phase =
  | 'setup'      // プレイヤー登録・設定
  | 'draw'       // スロット抽選
  | 'prepare'    // 準備時間
  | 'present'    // 発表タイマー
  | 'vote'       // パスプレイ秘密投票
  | 'result'     // 採択/不採択の発表
  | 'final';     // 最終ランキング

type Vote = 'accept' | 'reject';

type GameState = {
  phase: Phase;
  settings: Settings;
  players: Player[];          // { id, name, score, rerollsLeft }
  round: number;              // 現在の周回 (1-indexed)
  presenterIndex: number;     // players 内の添字
  hand: [Card, Card, Card] | null;   // 分野 / 手法 / 制約
  votes: Record<PlayerId, { vote: Vote; comment?: string }>;
  votingIndex: number;        // パスプレイで今誰が投票中か
  recentCardIds: string[];    // 直近3手番のカード。抽選から除外する
};
```

## 得点計算（唯一の真実の場所）

```ts
// 過半数 = 投票者数の半分より多い。同数は不採択。
const accepts = voters.filter(v => votes[v.id].vote === 'accept');
const accepted = accepts.length * 2 > voters.length;
const unanimous = accepted && accepts.length === voters.length;

presenter.score += unanimous ? 3 : accepted ? 2 : 0;
// 多数派に投票した査読者は +1（同数=不採択なので、この場合の多数派は reject 側とする）
for (const v of voters) {
  const votedAccept = votes[v.id].vote === 'accept';
  if (votedAccept === accepted) v.score += 1;
}
```

## テスト

- `game/` の純粋関数（得点計算・過半数判定・抽選の除外ロジック）に Vitest でユニットテストを書く。
