# 査読ゲーム 実装仕様

ゲームルール・カード設計・演出仕様の正本は、zikken リポジトリの
[`playground/peer-review-game-plan/`](https://github.com/tukiguti/zikken/tree/main/playground/peer-review-game-plan) に置く。
ルール変更は正本と実装を同じ変更単位で更新し、このリポジトリ内に仕様全文を複製しない。

## 実装上の不変条件

- 3〜8人。同じ名前・空欄の名前では開始できない。
- 1人の手番は `draw → prepare（設定時のみ）→ present → vote → result` の順に進む。
- カード構成は1〜5枚。各スロットを分野・手法・制約から選べ、同じ種類を複数指定できる。
- 各スロットは独立して `all`（おまかせ）・`serious`（真面目）・`neta`（ネタ）を選べる。抽選と引き直しは対象スロットの指定を守る。
- 初回の一括抽選は1手番につき1回だけ。以降はプレイヤーごとの残り回数を消費し、1枚ずつ引き直す。
- 同種カードを複数引いても同一カードは重複させない。絞り込み後の候補が足りない構成では開始できない。
- 分野・手法・制約の3デッキは、全ジャンルと serious/neta の組み合わせで必ず1枚以上ある。
- 直近3手番に出たカードは候補から外す。ただし候補が空になる場合だけ、そのデッキを復帰させる。
- 発表者は投票しない。査読者全員の投票完了前に結果表示・得点加算を行わない。
- Accept が投票者の半数を**超えた**場合だけ採択。同数は Reject 判定とする。
- 発表者は採択+2点、満場一致採択+3点（重複加算なし）。判定側と同じ票の査読者は+1点。
- 投票内容は各査読者の送信後に画面から消し、全員完了後だけ氏名付きで公開する。
- 設定保存に失敗してもゲーム本体を停止させない。
- 演出オフでも、リール中央と確定カードの内容を一致させる。

## 自動テストの必須範囲

- 得点: 同数Reject、通常採択、満場一致、査読者加点。
- 抽選: 直近カード除外、空デッキ時の復帰、ジャンル・スロット別tone絞り込み、引き直し時のtone維持。
- データ: 全カードIDの一意性、全モード組み合わせの非空性。
- 状態遷移: 無料再抽選の禁止、リロール上限、投票完了前の進行禁止、結果画面以外からの次手番禁止、8人×3周で全査読者が過不足なく投票すること。
- 可変構成: 1〜5枚、分野×3の重複なし抽選、スロット別の真面目・ネタ混在、同種カードのリロール、絞り込み後の候補不足判定。
- 設定: 入力途中の名前を巻き戻さない、空欄・重複名で開始しない、旧全体tone設定をスロット別設定へ移行する。

## 公開前チェック

```bash
npm ci
npm test
npm run build
```

続いて実ブラウザで、最低限次のシナリオを確認する。

1. 3人・1周・準備なし・演出オフで開始する。
2. 名前を一度空にして再入力し、フォーカスが外れず、空欄中は開始できないことを確認する。
3. 設定枚数の抽選後に `LEVER` が再利用できず、1枚引き直すと残数が減ることを確認する。
4. Accept 1票 / Reject 1票で不採択となり、Rejectした査読者だけ+1点になることを確認する。
5. 3人全員の手番を終え、最終ランキング・学会MVP・各称号を確認する。
6. 幅320px相当の画面でも横スクロールや操作不能がないことを確認する。
7. 同じ種類のスロットに真面目とネタを別々に設定し、抽選・引き直し後のカードがそれぞれの指定に合うことを確認する。
